# laravel-bfsg v2.2 — Ignore Mechanism + SARIF + Diff Reports

**Date:** 2026-04-09
**Status:** Design Spec — awaiting user review
**Target Release:** v2.2.0

## Overview

Three high-impact, independent features bundled into a single additive release:

1. **Ignore Mechanism** — Suppress false positives in specific elements via HTML comments, data attributes, or Blade directives.
2. **SARIF Output Format** — Export violations in SARIF 2.1.0 for GitHub Code Scanning and other CI tools.
3. **Diff Reports** — Compare two historical reports with precise per-violation matching.

All changes are backward compatible. No breaking changes. Public API surface only grows.

## Design Decisions Summary

| Decision | Choice | Rationale |
|---|---|---|
| Ignore scoping syntaxes | All three (comment next-sibling, comment block, data-attribute) | Each covers a distinct use case; shared parser code |
| Diff matching strategy | Hybrid fingerprint + grouping fallback | Precise for new reports, still usable for legacy reports |
| SARIF scope | Minimal (URL-based location, no Blade source linking) | Blade source linking requires separate infrastructure (Roadmap #7) |
| Ignore scope | HTML varieties + Blade directive, no config-based URL ignores | Blade directive is cheap extension of HTML comments; URL ignores belong to Rule Configuration (Roadmap #9) |

### Integration Points in Existing Code

- `src/Bfsg.php` — `analyze()` instantiates `IgnoreCollector` once per DOM and calls `$analyzer->setIgnoreCollector($collector)` on each registered analyzer that is a `BaseAnalyzer` instance (checked via `instanceof`). Analyzers that don't extend `BaseAnalyzer` are skipped — they continue to work unchanged. After analysis, `Bfsg::analyze()` also aggregates `$analyzer->getIgnored()` into a top-level `$this->ignored` array for `--show-ignored` output.
- `src/Analyzers/*.php` — Top-5 analyzers (`ImageAnalyzer`, `ContrastAnalyzer`, `FormAnalyzer`, `AriaAnalyzer`, `LinkAnalyzer`) extend new `BaseAnalyzer` and call `$this->shouldIgnore($element, $ruleKey)` before pushing violations. - `src/Reports/ReportGenerator.php` — New `case 'sarif'` in format `match`, delegates to `SarifGenerator`
- `src/Commands/BfsgHistoryCommand.php` — New `--diff`, `--from`, `--to` flags and `showDiff()` method
- `src/Models/BfsgViolation.php` — New `fingerprint` fillable field, auto-computed in `creating` model event
- `src/BfsgServiceProvider.php` — Registers Blade directive, publishes new migration

## Feature 1: Ignore Mechanism

### User-Facing Syntax

**Rule keys** are analyzer keys from `Bfsg.php`: `images`, `forms`, `headings`, `contrast`, `aria`, `links`, `keyboard`, `language`, `tables`, `media`, `semantic`, `page_title`, `input_purpose`, `focus`, `error_handling`, `status_messages`. The keyword `all` disables all analyzers for the marked scope. Multiple keys are comma-separated. Matching is case-insensitive, whitespace trimmed. Unknown keys are silently ignored to prevent version-drift breakage.

**Variant 1 — Next-sibling HTML comment:**

```html
<!-- bfsg-ignore:contrast -->
<div class="brand-hero">...</div>  <!-- ignored -->
<div class="other">...</div>        <!-- NOT ignored -->
```

**Variant 2 — Block HTML comment:**

```html
<!-- bfsg-ignore-start:contrast,aria -->
<section>...</section>
<section>...</section>
<!-- bfsg-ignore-end -->
```

Unclosed blocks are treated as implicit end-at-EOF with a verbose-mode warning.

**Variant 3 — `data-bfsg-ignore` attribute:**

Element itself AND all descendants inherit the ignore:

```html
<div data-bfsg-ignore="contrast">
  <p>Paragraph inherits ignore</p>
  <span>Span too</span>
</div>
```

Empty value (`data-bfsg-ignore=""`) ignores nothing.

**Variant 4 — Blade directive:**

```blade
@bfsgIgnore('contrast', 'images')
  <div>...</div>
@endBfsgIgnore
```

Compiles to `<!-- bfsg-ignore-start:contrast,images -->...<!-- bfsg-ignore-end -->` in rendered HTML. Uses Laravel's Blade compiler extension API registered in the service provider.

### IgnoreCollector Algorithm

Invoked once per `DOMDocument` at the start of `Bfsg::analyze()`. Builds an internal index keyed by DOMElement references:

```php
class IgnoreCollector
{
    /** @var \SplObjectStorage<\DOMElement, array<string>> */
    private SplObjectStorage $index;

    public function __construct(DOMDocument $dom)
    {
        $this->index = new SplObjectStorage();
        $this->collectComments($dom);
        $this->collectDataAttributes($dom);
    }

    public function isIgnored(DOMElement $element, string $ruleKey): bool
    {
        if (! $this->index->contains($element)) {
            return false;
        }
        $rules = $this->index[$element];
        return in_array('all', $rules, true) || in_array($ruleKey, $rules, true);
    }

    private function collectComments(DOMDocument $dom): void { /* ... */ }
    private function collectDataAttributes(DOMDocument $dom): void { /* ... */ }
}
```

**Pass 1 — Comments:** `DOMXPath::query('//comment()')` finds all HTML comments. For each:
- `bfsg-ignore:<keys>` without `-start/-end` → locate next element sibling (skipping text/whitespace nodes), add rule keys to its entry
- `bfsg-ignore-start:<keys>` → walk forward through following siblings and their descendants until a matching `bfsg-ignore-end` comment is found. All elements in between get the rule keys.

**Pass 2 — Data attributes:** `//*[@data-bfsg-ignore]` finds all marked elements. Parse the attribute value as comma-separated keys. Element itself plus all `descendant::*` get those keys merged into their entries.

**Collision handling:** If multiple sources target the same element, rule keys accumulate (union).

### Analyzer Integration

New `BaseAnalyzer` abstract class with a `reportViolation()` helper that routes violations to either the active list or the ignored list based on the collector lookup. This keeps analyzer call sites simple and preserves ignored violations for `--show-ignored` display:

```php
abstract class BaseAnalyzer
{
    protected array $violations = [];
    protected array $ignored = [];
    protected ?IgnoreCollector $ignoreCollector = null;

    public function setIgnoreCollector(?IgnoreCollector $collector): void
    {
        $this->ignoreCollector = $collector;
    }

    /**
     * Report a violation. Routes to the ignored list if the element+rule is
     * suppressed by the active IgnoreCollector; otherwise appends to violations.
     */
    protected function reportViolation(DOMElement $element, string $ruleKey, array $data): void
    {
        if ($this->ignoreCollector?->isIgnored($element, $ruleKey) ?? false) {
            $this->ignored[] = $data + ['_ignored_rule' => $ruleKey];
            return;
        }
        $this->violations[] = $data;
    }

    public function getIgnored(): array
    {
        return $this->ignored;
    }

    abstract public function analyze(DOMDocument $dom): array;
}
```

Top-5 analyzers migrated in this release: `ImageAnalyzer`, `ContrastAnalyzer`, `FormAnalyzer`, `AriaAnalyzer`, `LinkAnalyzer`. Each becomes:

```php
class ImageAnalyzer extends BaseAnalyzer
{
    public function analyze(DOMDocument $dom): array
    {
        $this->violations = [];
        $this->ignored = [];
        $xpath = new DOMXPath($dom);

        foreach ($xpath->query('//img[not(@alt)]') as $img) {
            $this->reportViolation($img, 'images', [
                'type' => 'error',
                'rule' => 'WCAG 1.1.1',
                'element' => 'img',
                'message' => 'Image without alt text found',
                'src' => $img->getAttribute('src'),
                'suggestion' => 'Add an alt attribute to describe the image',
                'auto_fixable' => true,
            ]);
        }
        // ... rest unchanged, all violation pushes go through reportViolation()

        return ['issues' => $this->violations];
    }
}
```

Remaining 11 analyzers continue to work without modification (no `BaseAnalyzer`, no ignore awareness). They push directly to `$this->violations` and are not affected by ignore markers. They are migrated incrementally in v2.3 as a non-breaking follow-up.

### Score Handling and `--show-ignored`

Ignored violations never enter the active `$this->violations` arrays in the first place — `BaseAnalyzer::reportViolation()` diverts them to `$this->ignored` at report time. After all analyzers run, `Bfsg::analyze()` aggregates each analyzer's `getIgnored()` output into a top-level `$this->ignored` collection. The score calculation operates on `$this->violations` only, so ignored items have zero effect on the grade.

CLI output:

```
✗ Found 12 violations (grade: C)
✓ Ignored 3 violations (hidden — use --show-ignored)
```

With `--show-ignored`:

```
Ignored violations (3):
  contrast | WCAG 1.4.3 | <div class="brand-hero">
    Contrast 2.1:1 below AA threshold
    → Ignored by: HTML comment (next-sibling)
```

## Feature 2: SARIF Output Format

### Schema and Version

SARIF 2.1.0 ([OASIS Standard](https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html)). Generated output validated in tests against `https://json.schemastore.org/sarif-2.1.0.json` using `justinrainbow/json-schema` (new dev dependency, ~200 KB, widely used).

### Rule Catalog

All 16 analyzers declared in `tool.driver.rules` with stable IDs of the form `bfsg/<analyzer_key>`. Rule entries are static data; lazy-loaded from a hardcoded array in `SarifGenerator` (not from config, to keep SARIF output deterministic across environments).

### Severity Mapping

| BFSG type | SARIF level |
|---|---|
| `error` | `error` |
| `warning` | `warning` |
| anything else | `note` |

### Fingerprint Reuse

Uses the same `ViolationFingerprint::compute()` function that powers Diff Reports. Writes the 16-char hex to `partialFingerprints["bfsgFingerprint/v1"]`. GitHub Code Scanning automatically deduplicates findings with matching partialFingerprints across workflow runs — "same finding as last run" is flagged, not re-reported.

### Integration in `ReportGenerator`

## Feature 3: Diff Reports

### Violation Fingerprint

Stable hash function that yields the same result for the same logical violation across runs:

```php
final class ViolationFingerprint
{
    public static function compute(array $violation): string
    {
        $normalized = implode('|', [
            $violation['analyzer'] ?? '',
            $violation['wcag_rule'] ?? '',
            $violation['element'] ?? '',
            self::normalizeMessage($violation['message'] ?? ''),
            $violation['src'] ?? '',
        ]);
        return substr(sha1($normalized), 0, 16);
    }

    private static function normalizeMessage(string $message): string
    {
        $message = preg_replace('/\d+(\.\d+)?/', '<num>', $message);
        $message = preg_replace('/https?:\/\/\S+/', '<url>', $message);
        return trim($message);
    }
}
```

Normalizing dynamic values (numbers, URLs) prevents fingerprint drift from cosmetic data changes. "Contrast 2.1:1 below threshold" and "Contrast 2.3:1 below threshold" receive the same fingerprint.

### Migration


### ReportDiffer

```php
final class ReportDiffer
{
    public function diff(BfsgReport $from, BfsgReport $to): DiffResult
    {
        $fromViolations = $from->violations;
        $toViolations = $to->violations;

        $canUseFingerprint = $fromViolations->every(fn ($v) => ! empty($v->fingerprint))
                          && $toViolations->every(fn ($v) => ! empty($v->fingerprint));

        return $canUseFingerprint
            ? $this->diffByFingerprint($fromViolations, $toViolations)
            : $this->diffByGrouping($fromViolations, $toViolations);
    }

    private function diffByFingerprint(Collection $from, Collection $to): DiffResult
    {
        $fromKeys = $from->keyBy('fingerprint');
        $toKeys = $to->keyBy('fingerprint');

        return new DiffResult(
            fixed: $fromKeys->diffKeys($toKeys)->values(),
            new: $toKeys->diffKeys($fromKeys)->values(),
            unchanged: $fromKeys->intersectByKeys($toKeys)->values(),
            mode: 'fingerprint',
        );
    }

    private function diffByGrouping(Collection $from, Collection $to): DiffResult
    {
        // Aggregate mode for legacy reports (no fingerprints). Groups
        // violations by (analyzer, wcag_rule, element) tuple and compares
        // counts per group. Individual violations cannot be attributed
        // across reports — we only report net deltas per group.
        $groupKey = fn ($v) => $v->analyzer.'|'.$v->wcag_rule.'|'.$v->element;
        $fromGroups = $from->groupBy($groupKey)->map->count();
        $toGroups = $to->groupBy($groupKey)->map->count();

        // The fixed/new/unchanged collections in grouping mode hold
        // synthetic "group delta" rows instead of real BfsgViolation rows.
        // The CLI renderer detects mode === 'grouping' and switches
        // output format accordingly.
        return new DiffResult(
            fixed: $this->groupDelta($fromGroups, $toGroups, subtract: true),
            new: $this->groupDelta($toGroups, $fromGroups, subtract: true),
            unchanged: collect(),
            mode: 'grouping',
        );
    }
}
```

### DiffResult DTO

```php
final readonly class DiffResult
{

        public Collection $fixed,
        public Collection $new,
        public Collection $unchanged,
        public string $mode,  // 'fingerprint' or 'grouping'
    ) {}
```


## Out of Scope for v2.2

Documented to prevent scope creep. Each items has a clear future home:

- **Config-based URL ignores** → Roadmap #9 "Rule Configuration"
- **Blade-source linking in SARIF** → Roadmap #7 "Blade Directive Scanner"
- **Analyzer migration for the remaining 11 analyzers** → v2.3 (non-breaking)
- **WCAG rule codes as ignore keys** (`bfsg-ignore:WCAG-1.4.3`) → v2.3 consideration
- **GitHub Action Marketplace listing** → Roadmap #4
- **Auto-fix suggestions for ignored violations** → Separate future feature

## Non-Goals

Explicit things this release does NOT try to do:

- Does not attempt to annotate Blade source files in SARIF output
- Does not add new analyzer types (no new WCAG coverage in v2.2)
- Does not refactor the remaining 11 analyzers
- Does not change the report persistence schema beyond the fingerprint column
- Does not introduce a new public Facade method; ignore/diff/sarif features are invoked through existing entry points (`Bfsg::analyze()`, `ReportGenerator`, `BfsgHistoryCommand`)
